# 云计算课程设计两人分工文档

## 1. 分工目标

本项目目标是完成课程设计任务书中的全部基础要求，并尽量完成所有附加题，争取取得高分。

两位同学的总体分工原则是：

- A 同学主要负责云环境、Kubernetes、CCE、部署、运维验证和云端截图。
- B 同学主要负责应用容器化、Spark 数据分析、PyTorch DDP、CI/CD 构建部分和实验报告主笔。
- A 的项目实施难度和环境调试成本更高，因此报告撰写部分适当减少。
- B 的代码分析和报告整理工作更多，用来平衡 A 的云环境工作量。
- 项目实施和报告撰写整体合并计算后，两位同学工作量约为 1:1。

## 2. 项目实施分工

| 任务 | 分值 | 负责人 | 具体完成内容 |
|---|---:|---|---|
| T1 应用容器化 | 10 | B | 修改后端 Dockerfile 和 requirements.txt，保留多阶段构建；修改前端首页，加入姓名和学号；本地运行 `docker compose up -d --build`；验证前后端通信；构建 backend/frontend 镜像并推送到 SWR；保存本地联调、后端返回、后端日志、SWR 镜像列表截图。 |
| T2 CCE 集群搭建 | 8 | A | 创建华为云 CCE 集群；配置 kubectl；确认 Worker 节点 Ready；执行并截图 `kubectl get nodes -o wide`，截图中需要包含 VERSION 列。 |
| T3 应用部署 | 12 | A | 编写和维护 backend Deployment、Redis Deployment、Service、ConfigMap、Secret 等 K8s YAML；后端副本数设置为 2，Redis 副本数设置为 1；配置 resources requests/limits；后端 Service 使用 LoadBalancer；通过 ELB 公网 IP 访问 `/api/ping` 并截图。 |
| T4 持久化存储 | 10 | A | 创建 Redis PVC；将 PVC 挂载到 Redis 的 `/data`；执行 Redis 写入测试；删除 Redis Pod 后重新查询数据；保存 `kubectl get pvc`、写入数据、删除 Pod、重建后查询数据的截图。 |
| T5 ConfigMap Volume 挂载 | 5 | A | 创建 Nginx 配置 ConfigMap；将 ConfigMap 以 volume 形式挂载到前端 Pod 的 `/etc/nginx/conf.d/default.conf`；修改 ConfigMap 后 exec 进入前端 Pod 验证配置文件变化；保存验证截图。 |
| T6 HPA 弹性伸缩 | 5 | A 负责 HPA，B 负责压测 | A 创建 backend HPA，监控 `kubectl get hpa` 和 `kubectl get pods -w`；B 使用 `ab`、curl 脚本或其他压测工具访问 `/api/ping`；两人共同保存扩容、缩容和压测截图。 |
| A-0 Spark Operator 环境部署 | 10 | A | 安装 Spark Operator；修改 SparkApplication YAML；提交 wordcount 或 Spark 分析作业；保存 Driver Pod 和 Executor Pod 运行状态截图。 |
| A-1 Spark 数据清洗 | 10 | B | 使用 PySpark 加载豆瓣电影数据集；打印 Schema 和前 5 行；统计各字段缺失值比例；至少对 2 个字段采用不同缺失值处理策略；输出清洗前后行数对比和 mean/std/min/max 等统计信息。 |
| A-2 Spark SQL 统计分析 | 15 | B | 完成至少 4 个 Spark SQL 或 DataFrame 查询，包含 GROUP BY 聚合、ORDER BY Top-N、时间维度趋势分析、JOIN 或窗口函数；每个查询保存结果截图，并准备不少于 50 字的分析说明。 |
| A-3 性能对比与 Amdahl 分析 | 5 | B | 分别使用 Pandas、PySpark executor=1、PySpark executor=2 运行同一类查询；记录执行时间；绘制性能对比图；结合 Amdahl 定律分析加速比未达到线性的原因。 |
| 附加题 1 监控系统 | +5 | A | 使用 Helm 部署 Prometheus + Grafana；配置或打开 Dashboard；截图节点 CPU 利用率、Pod 内存使用等指标；将截图和指标名称提供给 B 写入报告。 |
| 附加题 2 CI/CD 流水线 | +5 | A、B 共同 | B 负责流水线中的镜像构建、测试和推送 SWR；A 负责云端 K8s Deployment 镜像 Tag 更新和部署验证；两人共同保存流水线 Passed 截图、SWR 镜像更新截图、Deployment 镜像更新截图。 |
| 附加题 3 C-1 分布式 AI 训练 | +5 | B，A 协助云端运行 | B 负责 PyTorch DDP MNIST 代码、单机训练、分布式训练、训练时间对比和日志截图；A 协助在 K8s 上运行 PyTorchJob 或相关 Job；B 整理 AllReduce、数据并行、模型并行说明。 |

## 3. 报告撰写分工

报告部分由 B 同学作为主笔，A 同学主要提供云端实验材料。这样可以减轻 A 在环境部署和排错之外的负担，使两个人整体工作量更平衡。

| 报告内容 | 负责人 | 具体完成内容 |
|---|---|---|
| 封面、目录、整体格式 | B | 统一报告格式、标题层级、字体、图表编号、页码和最终 PDF 排版。 |
| 项目概述与分工说明 | B | 说明项目目标、总体架构、两人分工和工作量比例；A 负责确认分工描述是否准确。 |
| 华为云环境信息 | B 主写，A 提供材料 | A 提供 Region、CCE 版本、节点规格、节点截图；B 整理成报告文字。 |
| T1 应用容器化章节 | B | 写 Dockerfile 修改、本地联调、镜像构建、SWR 推送过程，并插入相关截图。 |
| T2 CCE 集群搭建章节 | B 主写，A 提供材料 | A 提供命令、截图和问题记录；B 整理 CCE 创建、kubectl 配置和节点 Ready 验证过程。 |
| T3 应用部署章节 | B 主写，A 提供材料 | A 提供 YAML、Pod、Service、ELB 访问截图；B 解释 Deployment、Service、ConfigMap、Secret 的作用。 |
| T4 持久化存储章节 | B 主写，A 提供材料 | A 提供 PVC、Redis 写入、删除 Pod、重查数据截图；B 写持久化验证过程和结论。 |
| T5 ConfigMap Volume 章节 | B 主写，A 提供材料 | A 提供 ConfigMap 挂载和 exec 验证截图；B 写 Volume 挂载与 envFrom 的适用场景区别。 |
| T6 HPA 弹性伸缩章节 | B 主写，A 提供材料 | A 提供 HPA、扩容、缩容截图；B 写扩容延迟、冷却时间和 HPA 降本价值分析。 |
| A-0 Spark Operator 章节 | B 主写，A 提供材料 | A 提供 Spark Operator、Driver Pod、Executor Pod 截图；B 整理 Spark on K8s 环境部署说明。 |
| A-1 数据清洗章节 | B | 写数据加载、Schema、前 5 行、缺失值比例、清洗策略、清洗前后行数对比和统计信息。 |
| A-2 SQL 统计分析章节 | B | 写 4 个查询的目的、SQL 或 DataFrame 逻辑、结果截图和每个查询不少于 50 字的分析。 |
| A-3 性能与 Amdahl 章节 | B | 写 Pandas 与 PySpark 性能表格、对比图、加速比和 Amdahl 定律分析。 |
| 附加题 1 监控系统章节 | B 主写，A 提供材料 | A 提供 Grafana Dashboard 截图；B 写 Prometheus Pull 采集原理和至少 3 个指标的含义。 |
| 附加题 2 CI/CD 章节 | B 主写，A 补充 | B 写流水线流程、镜像构建和推送过程；A 补充 K8s Deployment 更新验证；插入 Passed 和镜像 Tag 更新截图。 |
| 附加题 3 C-1 章节 | B | 写单机与分布式训练过程、训练时间对比、日志截图、AllReduce 梯度同步、数据并行与模型并行区别。 |
| 总结与收获 | B 初稿，A 补充 | B 写总体总结；A 补充云环境部署、K8s 排错和资源管理方面的经验。 |

| 同学 | 项目实施工作 | 报告工作 | 总体工作量 |
|---|---|---|---|
| A | 负责 CCE、K8s、PVC、ConfigMap Volume、HPA、Spark Operator、Prometheus/Grafana、CI/CD 云端验证等环境相关任务。环境依赖真实云平台，排错和截图成本较高。 | 不做报告主笔，主要提供命令、截图、问题记录和配置说明。 | 约 50%。项目实施更重，报告任务较轻。 |
| B | 负责 Docker、本地联调、SWR 推送、Spark 数据清洗、SQL 分析、性能对比、PyTorch DDP、CI/CD 构建部分。代码、图表和分析内容较多。 | 负责报告主笔、图表整理、文字分析、格式统一、最终 PDF 排版。 | 约 50%。项目实施较重，同时承担更多报告整理工作。 |

## 5. 可以写入报告的分工说明

本课程设计由两位成员共同完成，整体按照“云平台实施”和“应用数据分析及报告整合”进行分工。

A 同学主要负责华为云 CCE 集群、Kubernetes 应用部署、Redis 持久化、ConfigMap Volume 挂载、HPA 弹性伸缩、Spark Operator 部署、Prometheus/Grafana 监控以及 CI/CD 的云端部署验证。该部分依赖真实云环境，涉及 CCE、ELB、PVC、Helm、HPA、Operator 等组件，调试和截图验证成本较高。A 同学主要提供实验命令、关键截图、配置文件说明和问题记录。

B 同学主要负责应用容器化、本地 docker-compose 联调、镜像构建与推送、Spark 数据清洗、Spark SQL 统计分析、性能对比与 Amdahl 分析、PyTorch DDP 分布式 AI 训练以及实验报告主笔。B 同学同时根据 A 同学提供的云端实验材料，完成云平台相关章节的文字整理、原理分析、图表编号和最终 PDF 排版。

两位成员整体工作量约为 1:1。其中 A 同学侧重真实云环境搭建、部署验证和运维排错，B 同学侧重应用开发、并行计算实验、性能分析和报告整合。这样的分工既能保证项目任务并行推进，也能避免云环境调试和报告撰写同时集中在同一位同学身上。

## 6. 为什么这样分工

第一，云环境任务需要集中到一个人负责。T2、T3、T4、T5、T6、A-0 和监控系统都依赖 CCE、kubectl、Helm、Service、PVC、HPA 等云端组件。如果多人交叉修改 YAML 和集群资源，容易出现镜像地址、Service 名称、命名空间、端口、Secret 不一致的问题。因此这些任务由 A 同学统一负责更稳定。

第二，B 同学负责 Spark、DDP 和报告主笔，可以平衡 A 的环境压力。云环境部署通常耗时长、不可控，而 Spark 数据分析、Amdahl 分析、C-1 专题和报告排版也需要大量时间。让 B 承担更多分析文字和报告整理，可以使两个人的总工作量更接近。

第三，这样分工更适合高分验收。A 同学专注保证云端功能真实跑通，截图完整，包括 Pod Running、PVC Bound、HPA 扩缩容、Grafana Dashboard 等。B 同学专注保证报告内容完整，分析充分，包括 Spark 查询分析、性能图表、Amdahl 解释、AllReduce 原理和总结反思。这样既能保证实验功能不漏项，也能提高报告质量分。

第四，这样便于答辩说明。A 同学可以重点讲清楚云平台、K8s、PVC、HPA 和监控系统；B 同学可以重点讲清楚 Docker、Spark 数据分析、性能对比、DDP 和报告分析。两个人各自负责的技术线清晰，不容易出现答辩时互相解释不清的问题。
