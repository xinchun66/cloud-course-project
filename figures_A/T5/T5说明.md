# T5 ConfigMap Volume 挂载说明

## 完成状态

T5 已完成。当前已使用 `frontend-nginx-config` ConfigMap 保存 Nginx 反向代理配置，并通过 Volume 形式挂载到 frontend Pod 的 `/etc/nginx/conf.d/default.conf`。随后修改 ConfigMap 内容，重启 frontend Deployment 后进入新 Pod 查看配置文件，确认更新后的配置已生效。

## 资源配置

| 资源 | 名称 | 说明 |
|---|---|---|
| ConfigMap | `frontend-nginx-config` | 保存 Nginx `default.conf` 配置。 |
| Deployment | `frontend` | 将 `frontend-nginx-config` 以 Volume 形式挂载到 Nginx 配置路径。 |
| 挂载路径 | `/etc/nginx/conf.d/default.conf` | Nginx 默认站点配置文件。 |

当前 frontend Deployment 中的挂载关系为：

```text
/etc/nginx/conf.d/default.conf <- nginx-config/default.conf <- frontend-nginx-config
```

## 执行命令

查看 Nginx ConfigMap 内容：

```bash
kubectl describe configmap frontend-nginx-config
```

查看 frontend Deployment 的 Volume 挂载情况：

```bash
kubectl describe deploy frontend
```

进入 frontend Pod 查看默认 Nginx 配置文件：

```bash
FRONTEND_POD=$(kubectl get pod -l app=frontend -o jsonpath='{.items[0].metadata.name}')
kubectl exec $FRONTEND_POD -- cat /etc/nginx/conf.d/default.conf
```

修改 ConfigMap 内容，在 Nginx 配置中加入验证标记：

```bash
cat > /tmp/default.conf <<'EOF'
server {
    listen 80;

    # T5 ConfigMap volume update verified

    location / {
        root /usr/share/nginx/html;
        index index.html;
    }

    location /api/ {
        proxy_pass http://backend-svc:80;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
EOF

kubectl create configmap frontend-nginx-config \
  --from-file=default.conf=/tmp/default.conf \
  -o yaml --dry-run=client | kubectl apply -f -
```

重启 frontend Deployment，使新 Pod 重新挂载更新后的 ConfigMap：

```bash
kubectl rollout restart deployment frontend
kubectl rollout status deployment frontend
```

再次进入 frontend Pod 验证配置文件内容：

```bash
FRONTEND_POD=$(kubectl get pod -l app=frontend -o jsonpath='{.items[0].metadata.name}')
kubectl exec $FRONTEND_POD -- cat /etc/nginx/conf.d/default.conf
```

## 说明

本次 frontend Deployment 使用了 `subPath` 将 ConfigMap 中的 `default.conf` 单文件挂载到 Nginx 配置路径。由于 `subPath` 挂载的文件不会稳定地自动感知 ConfigMap 更新，因此修改 ConfigMap 后通过 `kubectl rollout restart deployment frontend` 创建新的 frontend Pod，让新 Pod 在启动时重新挂载最新 ConfigMap 内容。该方式不影响 T5 的验证目标，能够清楚证明 Nginx 配置来自 ConfigMap Volume，并且修改后的配置已进入 Pod 内文件系统。

## 结论

本任务完成了 Nginx 配置的 ConfigMap Volume 挂载验证。修改 `frontend-nginx-config` 后，frontend Pod 中的 `/etc/nginx/conf.d/default.conf` 能够看到更新后的配置内容，说明 ConfigMap 以 Volume 形式提供配置文件的机制生效。
